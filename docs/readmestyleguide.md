# Systems Narrative Architecture: Writing Guardrails

## Operational Pipeline
This document functions as a transformation and processing contract for manuscript preparation and technical editing. It converts unstructured conceptual or narrative drafts into structured documentation while protecting semantic integrity and authorial voice. 

### Execution Priority Stack
When semantic constraints or stylistic choices conflict during processing, the processing system must enforce the following absolute priority ranking:
1. **Preserve:** Retain the kinetic, energetic rhythm and structural momentum of the author’s conceptual interface language. Do not flatten individual personality into generic corporate prose.
2. **Normalize:** Ground all underlying analytical assertions by enforcing precise mathematical, complexity-theory, and statistical terminology.
3. **Transform:** Reconstruct unstructured or non-linear raw drafts into a clear, cascading systems hierarchy.

### Target Output Context
Unless otherwise specified, the default output target must compile into structured, concise technical documentation optimized for public repository deployment and professional portfolio validation.

---

## Structural Architecture

Documentation should structure information hierarchically to mirror the underlying system design:
* **Direct Mapping First:** Lead immediately with the primary technical mechanism or problem statement within the introductory layer. Remove purely redundant exposition that does not contribute directly to structural clarity or narrative voice.
* **Three-Tier Functional Layout:**
  * **System Overview:** Defines the conceptual boundary, the problem constraint, and the reduction strategy.
  * **Technical Parameters:** Maps the explicit boundaries, input/output constraints, and formal theorem anchors.
  * **Core Logic Flow:** Provides the structural breakdown detailing how the implementation resolves the complexity scaling.

---

## The Structural-Narrative Interface

Maintaining documentation credibility requires managing the transition between the conceptual interface and the mathematical substrate.

* **Preferred Semantic Metaphors (The Interface):** Utilized to convey the movement, behavior, and structural transformation of the logic. Metaphorical vocabulary here should preserve active narrative energy. Preferred metaphors include concepts of collapse, compression, mirroring, cascading, pipeline, and trajectory.
* **Analytical Grounding (The Substrate):** Utilized strictly when asserting mathematical, algorithmic, or complexity-theory properties. Target lexicon focuses on formal terms like induction, partition, complementarity, asymptotic bound, and constant-time execution per step under scaling.

*Strategic Guardrail:* Avoid letting the interface layer accidentally make unverified analytical claims (e.g., using descriptive velocity terms when describing an implementation, rather than defining the explicit complexity class in the analytical substrate).

---

## Semantic Stability & Anchoring

To eliminate inflation drift without flattening individual voice, enforce these diagnostic checks before final text output:

* **Alternative Framing Consolidation:** Establish a single canonical narrative unit for metrics and data scale at the outset of the document and enforce it universally (e.g., utilize "101-digit integers" consistently; do not shift between alternative mathematical notations within the same file).
* **Soft Constraint Bounding:** Replace unverified runtime metrics (e.g., "executes in milliseconds") with bounded constraint language unless accompanied by an explicit hardware benchmark profile. Let complexity theory carry the performance claim: *"reducing runtime to negligible practical scales for typical inputs."*
* **Theorem Contextualization:** Ground primary structural assertions by identifying the underlying framework within the sentence structure via parenthetical annotation: *"...which induces a complete, non-overlapping partition (via Beatty complementarity)."*
